# Hydra 2.0 "Core" — Design Spec

**Date:** 2026-04-17
**Status:** approved (brainstorming → writing-plans)
**Milestone target:** 4–6 weeks
**Author:** Franz + Claude (via superpowers:brainstorming)
**Supersedes:** Phase 2 plan in `hydra-next-gen-roadmap.md` (sections "Phase 2: Intelligence")

---

## 1. Motivation

Hydra 1.x delivered a working multi-advisor council with structured JSON epilogs, confidence calibration, and smart context windowing (Phase 1 Foundation, merged 2026-04-16). Four weaknesses remain:

1. **Hallucinated findings.** Advisors occasionally cite file ranges that don't exist, misremember line numbers, or assert patterns a reader can disprove in 10 seconds.
2. **Overlap & incompleteness.** The current roster has Mies + Stranger as two near-adjacent advisors while core review dimensions (PR-description/intent consistency, phantom helpers, over-engineering, fake-TDD) are uncovered.
3. **Token waste in fan-out.** Each of 3–6 advisors re-reasons from the raw diff in parallel — much of that work is duplicated (every advisor re-discovers the same CVE-flagged dependency, the same broken import).
4. **No measurable proof of improvement.** Releases feel better but cannot be compared across versions. Competitors (CodeRabbit, Greptile, Vercel Agent) have tool-use grounding + benchmarks; Hydra does not.

**2.0 Core addresses all four** without expanding scope into learning/memory (deferred to 2.1) or new model families (deferred to 2.2).

---

## 2. Non-Goals (explicit scope fences)

The following are **deliberately out of scope** for 2.0 Core. Any temptation to include them should be rejected:

- **Finding Disposition Memory / Playbooks / Lessons Log / Conventions** → Hydra 2.1 "Memory"
- **Pre-Review Inquiry Loop** → evaluated, deferred; optional `--ask` flag only
- **Advisor Playbook evolution / prompt self-editing** → Hydra 2.2+
- **Advisor Accuracy Meta-Model** → needs months of history; 2.2+
- **Gemini / Grok / DeepSeek / Qwen integration** → 2.2+ pending verified 2026 API research
- **Sandbox-verified council patches** → separate milestone
- **Sentinel split (AuthZ/Data), new advisors Scribe/Proctor, conditional advisors (Braille/Polyglot/Lab)** → decide based on real usage data in 2.1+
- **Public benchmark submission (Martian)** → internal bench only in 2.0; submit after 2.0 stabilizes with real numbers
- **Cross-project knowledge sharing** → privacy-design needed first
- **Learning-from-feedback loops of any kind** → 2.1

---

## 3. In-Scope Features (7 total)

1. **Grounding Layer** — Mechanical Citation Verification (always on) + Skeptic-Pass (`--verify` opt-in)
2. **Two-Phase Pipeline** — Deterministic seed (tools + Echo + Navigator-in-deep) → LLM advisors extend/disagree
3. **Echo Advisor** — Claude-Code self-review: phantom helpers, over-engineering, fake-TDD, plan-vs-diff drift
4. **Mies+ Fusion** — merges current Mies (subtractive) + Stranger (readability) into one cross-model advisor
5. **`--tensions` Flag** — filter output to disagreements between advisors
6. **Per-Agent Model Overrides** — `--agent <role>=<model>` CLI + config-file support
7. **Benchmark Infrastructure** — `.hydra/benchmark/` local runner + 10 starter cases (internal only, no public submission)

Each feature is specified in full below.

## 4. Architecture Overview

### 4.1 Pipeline (new two-phase flow)

```
Phase 0: Pre-flight Gate                           [existing, unchanged]
  - Concrete/Hydra-worthy/size/secrets checks
  - Mode resolution: standard | deep
  - Profile resolution: quality | balanced | budget (from model-profiles spec)
  - Per-agent override resolution (NEW, see §9)

Phase 1: Deterministic Seed                        [NEW]
  Tools (parallel, optional — graceful degrade if not installed):
    - Semgrep (p/default + p/security + p/secrets)
    - OSV-Scanner (lockfile → CVEs)
    - Lang-Checker (auto-detect tsc | mypy | ruff | eslint)
  LLM seeders (parallel):
    - Echo Advisor (Opus)         [always, standard+deep]
    - Navigator Advisor (Sonnet)  [deep only; existing advisor, new placement]
  ──────────────────────────────────────
  Output: seed_report.json  (schema §10.1)

Phase 2: LLM Advisor Council                       [upgraded]
  Advisors receive: diff + seed_report + semi-formal prompt (Premise→Trace→Conclusion)
  Each finding must tag:
    - extends_seed: <id>       (new finding building on a seed)
    - challenges_seed: <id>    (disputes a seed finding)
    - novel: true              (no seed connection)
  Standard: Cassandra (Opus) + Mies+ (Codex) + Sentinel (Codex)
  Deep:     + Volta (Sonnet) + Peer-Review Phase (existing)
  (Navigator runs ONLY in Phase 1 in deep mode; no Phase-2 re-pass.
   Other advisors receive Navigator's structural map via seed_report.)

Phase 2.5: Grounding Pass                          [NEW]
  Step 1 — Mechanical Citation Verify (always on, deterministic):
    For each finding with file:start-end
      - Read range from disk
      - Fuzzy-match claimed token/pattern present in range
      - Miss → tag GROUNDING_FAIL, demote to UNGROUNDED
  Step 2 — Skeptic-Pass (opt-in via `--verify`, Haiku):
    For each VERIFIED/HYPOTHESIS-HIGH finding that passed Step 1
      - Haiku receives ONLY: finding text + cited range + read_file tool
      - Must emit PASS | FAIL | UNCLEAR (structured)
      - FAIL → demote to UNVERIFIED; Chairman sees degradation note

Phase 3: Chairman Synthesis                        [upgraded]
  Receives: grounded findings (with is_tension flag, see §8)
  Must emit: verdict + confidence (existing) + tensions section (NEW)

Phase 4: Outputs                                   [upgraded]
  Standard report:
    + Tensions section (if any tensions present)
    + Grounding summary (N findings, N mechanically verified, N skeptic-failed)
  `hydra this --tensions`: output filtered to tension findings only
  `hydra bench`: separate workflow (see §11)
```

## 5. Feature 1 — Grounding Layer

### 5.1 Mechanical Citation Verification (always on)

**Input:** Advisor JSON epilog (one per advisor) after Phase 2.

**Algorithm (deterministic, no LLM calls):**

```
for each finding in epilog.findings:
  if not finding.file or not finding.lines:
    finding.grounding = "NO_CITATION"
    continue

  if not file_exists(finding.file):
    finding.grounding = "FILE_MISSING"
    demote_severity(finding)  # one level
    continue

  range_text = read_range(finding.file, finding.lines)
  if range_text is None:
    finding.grounding = "RANGE_MISSING"
    demote_severity(finding)
    continue

  claimed_tokens = extract_salient_tokens(finding.chain.code_construct)
  present_count = count_present(claimed_tokens, range_text)
  if present_count / len(claimed_tokens) < 0.4:
    finding.grounding = "TOKEN_MISMATCH"
    demote_severity(finding)
    continue

  finding.grounding = "MECH_VERIFIED"
```

**`extract_salient_tokens`:** identifiers, literal strings, function names from `chain.code_construct` (split on whitespace/punctuation, filter stopwords like "the/and/in", require ≥3 chars). Max 8 tokens.

**`demote_severity`:**
- SERIOUS → MODERATE
- MODERATE → MINOR
- MINOR → TRIVIAL
- VERIFIED evidence tag → HYPOTHESIS

**Output:** Same findings with added `grounding` field. Chairman receives all findings but instructed to weight `MECH_VERIFIED > HYPOTHESIS_LIKE > UNGROUNDED`.

**Grounding summary:** emitted to report regardless of mode:
```
## Grounding Summary
- Mechanical verification: 14/17 findings passed (82.4%)
- Failed: 2 TOKEN_MISMATCH, 1 FILE_MISSING → auto-demoted
```

### 5.2 Skeptic-Pass (`--verify`, opt-in)

**Trigger:** CLI flag `--verify` on `hydra this`. Also implicit default when running `hydra bench` against ground-truth cases.

**Model:** Haiku 4.5 (cheap, fast). Hardcoded for this role — not affected by `--profile` or `--agent chairman=X`. Rationale: skeptic must be a *different* reasoning pass; user downgrading is fine but user upgrading to Opus defeats the purpose (cost explosion without quality gain).

**Input per finding (passed MECH_VERIFIED only):**
- Finding title, severity, evidence tag, hypothesis text
- Cited range (file:start-end, raw text inlined — not a tool call)
- `read_file` tool available (read other files by path, for when the claim requires cross-file context)

**Prompt (stored in `references/skeptic-protocol.md`):**
```
You are a skeptic verifier. Your job is NOT to find new bugs.
Your job is to check whether THIS specific claim is supported by THIS code.

You have no memory of how the claim was made.
You see only the claim, the cited code, and can read other files if needed.

Emit exactly one JSON line:
{"verdict": "PASS" | "FAIL" | "UNCLEAR", "reason": "<≤30 words>"}

PASS = the cited code clearly exhibits the claimed issue.
FAIL = the cited code does NOT exhibit the claimed issue.
UNCLEAR = insufficient evidence to decide without running code.
```

**Budget:** max 5 `read_file` tool calls per finding, hard cap. 60-second wall-clock timeout per finding. On timeout or budget exhaustion → finding marked UNCLEAR.

**Output demotion:**
- FAIL → finding marked `SKEPTIC_REJECTED`, removed from chairman input (but listed in degradation panel)
- UNCLEAR → finding passes through with `grounding = "UNCLEAR"` annotation
- PASS → finding promoted to `grounding = "SKEPTIC_VERIFIED"`

**Cost estimate:** ~$0.001–0.003 per finding with Haiku. For a typical 15-finding review: +$0.015–0.045 (≈+10–15% of standard mode cost).

## 6. Feature 2 — Two-Phase Pipeline

### 6.1 Phase 1 — Deterministic Seed

Runs all tools **in parallel**. Each tool has a 60-second timeout. All tool outputs are unified into `seed_report.json` (§10.1) and injected as `[SECTION:seed_findings]` in Phase-2 advisor prompts.

**Semgrep:**
```bash
semgrep --config=p/default --config=p/security --config=p/secrets \
        --json --quiet --timeout=60 \
        --include=<changed_files> <repo_root>
```
- Required rulesets: `p/default`, `p/security`, `p/secrets`
- Findings filtered to changed files only (diff-scoped)
- Not installed → emit `[Hydra] Semgrep not installed. Phase 1 seed reduced (~15% quality drop). Install for higher fidelity: brew install semgrep`

**OSV-Scanner:**
```bash
osv-scanner --lockfile=<auto-detect> --json --output <tmpfile>
```
- Auto-detect: `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`, `requirements.txt`, `poetry.lock`, `go.sum`, `Cargo.lock`, `Gemfile.lock`
- Runs only if lockfile in changed files OR if it's the first Hydra run in this repo (one-time baseline)
- Not installed → skip silently (logged in state.json but no user-visible warning)

**Lang-Checker (auto-detect):**
- `tsc --noEmit` if `tsconfig.json` present and `*.ts`/`*.tsx` in changed files
- `mypy --no-error-summary` if `mypy.ini`/`pyproject.toml`/`*.py` in changed files
- `ruff check --output-format json` if `ruff` installed and `*.py` in changed files
- `eslint --format json` if `.eslintrc*` present and `*.js`/`*.ts` in changed files
- Short-circuit escape: if lang-checker reports **syntax/type errors** (not just warnings), Hydra emits `[Hydra] Source fails type-checking. Fix these first; aborting review: <top 3 errors>` and exits with code 2.

**Echo Advisor:** See §7.

**Navigator Advisor (deep mode only):** Existing advisor, unchanged prompt logic, new placement in Phase 1. Receives: full diff + repo file tree + PR description. Produces seed findings about architecture, boundaries, circular deps. **Does NOT run a Phase-2 re-pass in 2.0.** Phase-2 advisors receive Navigator's output via `seed_report.navigator_findings`. This simplifies the pipeline and cuts ~2k tokens from deep mode vs 1.x.

### 6.2 Phase 2 — Extend-or-Disagree protocol

Every Phase-2 advisor prompt now includes:

```
[SECTION:seed_findings]
<seed_report.json>
[/SECTION]

[INSTRUCTION]
The above findings were produced by deterministic tools, Echo, and
(in deep mode) Navigator. For each of your findings, you MUST emit
one of:
  - "extends_seed: <seed_id>"    → you're adding depth/context to an existing seed
  - "challenges_seed: <seed_id>" → you disagree with a seed finding
  - "novel: true"                → your finding has no seed connection

Findings that merely restate a seed finding will be rejected.
[/INSTRUCTION]
```

**Enforcement:** Orchestrator post-processes advisor JSON epilog. Findings without one of the three tags are **dropped** (logged in degradation panel, not surfaced to chairman). This is strict — the point is to force genuine contribution beyond the seed.

### 6.3 Token economics verification

Two-phase saves tokens because:
- Seed findings (often 5–15 items) carry ~800–1500 tokens
- Without seed, each of N advisors re-discovers them independently → N × waste
- With seed, advisors see facts once, spend tokens on reasoning beyond them

**Expected saving:** 25–35% in Phase-2 advisor inputs. Offsets Echo's +5–10% in standard mode (net ~0), nets –10% in deep mode (Echo + Navigator seed phase is cheap vs. the savings on 4 Phase-2 advisors + 3 reviewers).

**Verification (§11):** bench cases record token usage per mode; 2.0 must not exceed 1.x baseline token usage on the 10 starter cases.

## 7. Feature 3 — Echo Advisor

### 7.1 Role

Echo is the first advisor whose primary input is **the PR description + diff together**, not just the diff. It looks for failures modes specific to AI-assisted development (especially Claude Code's known patterns).

### 7.2 Inquiries (the 7 checks)

1. **Phantom helpers:** New utility/helper introduced in diff, called 0 or 1 times.
2. **Over-engineering:** Abstraction factories / config knobs / wrapper classes with one caller or one implementation.
3. **Fake TDD:** Tests that assert on implementation details (method call counts, private state) rather than observable behavior; tests clearly written after implementation (same-commit additions whose assertions mirror the code structure 1:1).
4. **Plan-vs-diff drift:** PR description promises X; diff delivers Y or Y'. Flag mismatch.
5. **Scope creep:** Files touched outside the stated task's natural boundary.
6. **Comment bloat:** Comments that narrate *what* the code does (duplicating well-named identifiers) rather than *why*.
7. **Defensive theatre:** `try/except` that swallows errors silently, null checks after guaranteed-non-null, unreachable defensive branches.

### 7.3 Prompt (lives in `references/echo-protocol.md`)

Echo emits the same structured JSON epilog as other advisors, but with two additional optional fields per finding:

```json
{
  "...": "...",
  "check_type": "phantom_helper | over_engineering | fake_tdd | drift | scope_creep | comment_bloat | defensive_theatre",
  "pr_desc_quote": "<relevant quote from PR description>"  // if drift/scope_creep
}
```

### 7.4 Where it runs

Phase 1 (seed phase) — Echo's findings become seeds that Phase-2 advisors can extend or challenge. This is deliberate: Echo's meta-level findings often point Phase-2 advisors at the right code locations.

**Model:** Opus by default (meta-reasoning is hard; cheap models miss the subtle checks). Overridable via `--agent echo=<model>`.

### 7.5 Edge case: no PR description

If invoked outside a PR context (local `hydra this` on uncommitted diff), Echo skips checks 4 and 5 (drift/scope_creep) and runs the remaining 5 checks. Emits a note in the report's grounding summary: `Echo: 5/7 checks active (no PR context)`.

## 8. Feature 4 — `--tensions` Flag + Tensions Detection

### 8.1 Tension definition (deterministic)

A finding is flagged `is_tension: true` if **any** of:

1. **Position split:** Two or more advisors produced correlated findings (same `file` + overlapping `lines` ±10 + same `issue_class` where defined), and at least one had `position: APPROVE` (= "this is fine") while another had `position: CONCERN` or `REJECT`.
2. **Severity delta ≥ 2:** Same correlated finding set has a severity range spanning ≥2 levels (e.g., one TRIVIAL, one SERIOUS).
3. **Cross-family split:** Two advisors from different model families (Anthropic vs Codex) disagree on position OR severity on the same correlated finding set.

### 8.2 Finding correlation algorithm

Simple, deterministic:
```
two findings F1 and F2 correlate if:
  F1.file == F2.file AND
  range_overlap(F1.lines, F2.lines, tolerance=10) AND
  F1.issue_class == F2.issue_class  # if both have issue_class tagged
  OR
  F1.file == F2.file AND range_overlap(...) AND title_keyword_overlap(F1, F2) ≥ 0.5
```
(Second clause: Jaccard similarity on title-word sets, threshold 0.5.)

Correlated findings are **grouped** into a single tension record:
```json
{
  "tension_id": "T-3",
  "findings": ["C-1", "M-2", "S-1"],
  "reasons": ["position_split", "cross_family_split"],
  "positions": {"Cassandra": "CONCERN", "Mies+": "APPROVE", "Sentinel": "REJECT"},
  "severities": {"C-1": "MODERATE", "M-2": "TRIVIAL", "S-1": "SERIOUS"}
}
```

### 8.3 Chairman's responsibility

Chairman receives tension records alongside findings. In the standard report, chairman emits a new section:

```
## Tensions
(3 findings where advisors disagreed — worth your attention)

### T-3: auth/middleware.ts:47-62 — token refresh concurrency
- Cassandra (Opus, CONCERN, MODERATE): race under concurrent calls
- Mies+ (Codex, APPROVE, TRIVIAL): suggests current design is fine
- Sentinel (Codex, REJECT, SERIOUS): flags potential session hijack
Chairman's read: The disagreement hinges on whether refreshToken() is
reentrant. Sentinel's concern stands if sessions can be hijacked
cross-request, which depends on unseen code in session.ts.
→ Recommend: investigate session.ts:refreshLock before merging.
```

### 8.4 `--tensions` flag behavior

`hydra this --tensions` emits **only** the tensions section (no verdict, no other findings). Useful when chairman's verdict is clear but user wants to focus on disagreements.

Combines with other flags: `hydra this --deep --tensions`, `hydra this --verify --tensions` etc.

## 9. Feature 5 — Per-Agent Model Overrides

### 9.1 CLI syntax

```bash
hydra this --agent cassandra=sonnet --agent mies=haiku
hydra this --deep --agent sentinel=opus --agent echo=sonnet
hydra this --agent chairman=sonnet    # emits downgrade warning
```

Flag is **repeatable**. Role names — canonical is underscored for JSON/config safety, aliases accepted:

| Canonical | Aliases |
|---|---|
| `cassandra` | — |
| `mies_plus` | `mies+`, `mies` (legacy) |
| `sentinel` | — |
| `volta` | — |
| `navigator` | `nav` |
| `echo` | — |
| `chairman` | `chair` |
| `reviewer` | `reviewers` (applies to all 3 deep-mode reviewers) |

Model values: `opus` (Opus 4.7), `sonnet` (Sonnet 4.6), `haiku` (Haiku 4.5), `codex` (whatever Codex runtime is configured). Full model names like `opus-4-7` also accepted.

### 9.2 Config file

`.hydra/config.json` (per-project, optional). Example:

```json
{
  "profile": "balanced",
  "agents": {
    "cassandra": "opus",
    "mies_plus": "codex",
    "sentinel":  "codex",
    "volta":     "sonnet",
    "navigator": "sonnet",
    "echo":      "opus",
    "chairman":  "opus"
  }
}
```

Keys may be omitted — omitted keys fall through to `profile` preset.

Also supports `~/.hydra/config.json` for user-global defaults. Per-project overrides user-global.

### 9.3 Precedence chain (highest wins)

1. `--agent <role>=<model>` CLI flag (one-time)
2. `.hydra/config.json` `agents.<role>` (per-project)
3. `~/.hydra/config.json` `agents.<role>` (user-global)
4. `--profile <preset>` CLI flag
5. `.hydra/config.json` `profile` (per-project)
6. `~/.hydra/config.json` `profile` (user-global)
7. Default preset `quality`

### 9.4 Chairman protection

If the resolved model for `chairman` is not Opus (any family), Hydra emits:
```
[Hydra] Warning: Chairman resolved to <model>. Chairman downgrade may
        degrade verdict quality. Chairman is the single point of failure
        for synthesis. Proceed? [Y/n]
```
User must confirm (non-interactive contexts: `HYDRA_ALLOW_CHAIRMAN_DOWNGRADE=1` skips the prompt).

### 9.5 Natural-language parsing

In conversational invocations (e.g. user types `Run Hydra with Cassandra on Sonnet and Mies on Codex`), Hydra's NL resolver must translate to canonical flags:
- "cassandra on <model>" → `--agent cassandra=<model>`
- "all advisors on sonnet" → `--agent cassandra=sonnet --agent mies+=sonnet --agent sentinel=sonnet ...`
- "budget but echo opus" → `--profile budget --agent echo=opus`

Parsing rules documented in SKILL.md step 0.9 (to be added).

### 9.6 `hydra profile show`

Command prints resolved model map after precedence application:
```
Profile: balanced (from .hydra/config.json)
Overrides from config: echo=opus, chairman=opus
Overrides from CLI: (none)

Resolved:
  Phase 1 seeders:
    echo       : opus-4-7
    navigator  : sonnet-4-6 (deep only)
  Phase 2 advisors:
    cassandra  : opus-4-7
    mies+      : codex
    sentinel   : codex
    volta      : sonnet-4-6 (deep only)
  Chairman     : opus-4-7
  Skeptic      : haiku-4-5 (--verify, hardcoded)
```

## 10. Data Contracts

### 10.1 `seed_report.json` schema

```json
{
  "schema_version": "2.0",
  "generated_at": "2026-04-17T14:30:00Z",
  "tool_findings": [
    {
      "id": "T-SEM-1",
      "source": "semgrep",
      "rule_id": "javascript.lang.security.audit.detect-eval-with-expression",
      "file": "src/util.js",
      "lines": "47-47",
      "severity": "MODERATE",
      "message": "Detected eval() with expression",
      "raw": { /* original tool output */ }
    }
  ],
  "echo_findings": [ /* normal advisor finding schema + check_type + pr_desc_quote */ ],
  "navigator_findings": [ /* normal advisor finding schema; deep mode only */ ],
  "skipped_tools": ["osv-scanner"],
  "warnings": ["Semgrep not installed: reduced fidelity"]
}
```

### 10.2 Extended advisor finding schema (2.0)

Adds three optional fields to existing 1.x schema:

```json
{
  "id": "C-1",
  "title": "Race in token refresh",
  "severity": "SERIOUS",
  "evidence": "VERIFIED",
  "position": "CONCERN",
  "file": "auth/middleware.ts",
  "lines": "47-62",
  "issue_class": "race_condition",
  "chain": {
    "premise": "single concurrent request assumed",
    "execution_trace": "req1 → refreshToken() → ... req2 → refreshToken()",
    "conclusion": "second call overwrites first token"
  },
  "extends_seed": "T-SEM-3",        // NEW (2.0)
  "challenges_seed": null,          // NEW (2.0)
  "novel": false,                   // NEW (2.0)
  "grounding": "MECH_VERIFIED",     // NEW (2.0, set by orchestrator not advisor)
  "is_tension": false               // NEW (2.0, set by orchestrator)
}
```

**Chain upgrade (Meta semi-formal format):** The existing `chain` object's 5 fields (file_line / code_construct / assumption / failure_mode / impact) are replaced by 3 explicit reasoning fields: `premise`, `execution_trace`, `conclusion`. This matches Meta's structured-prompting research (93% accuracy boost from format alone). Advisor prompts in `references/advisors.md` must be updated accordingly.

**Backward-compat:** 1.x state.json files (missing new fields) can be read by 2.0 orchestrator with sensible defaults (`novel: true`, `grounding: "UNKNOWN"`, `is_tension: false`).

### 10.3 State.json v3 schema

`state.json` evolves from v2 to v3. New fields:

```json
{
  "schema_version": "3.0",
  "seed_report": { /* see §10.1 */ },
  "grounding_summary": {
    "total_findings": 17,
    "mech_verified": 14,
    "mech_demoted": 3,
    "skeptic_ran": true,
    "skeptic_passed": 12,
    "skeptic_failed": 1,
    "skeptic_unclear": 1
  },
  "tensions": [ /* tension records, see §8.2 */ ],
  "resolved_models": { /* see §9.6 output */ },
  "token_usage": {
    "phase_1": { "tools_ms": 2340, "echo_tokens": 1850, "navigator_tokens": 0 },
    "phase_2": { "cassandra_tokens": 2100, "mies_plus_tokens": 1950, "sentinel_tokens": 1880 },
    "phase_2_5": { "skeptic_tokens": 450 },
    "phase_3": { "chairman_tokens": 1100 }
  }
}
```

Migration: 2.0 orchestrator reads v2 state.json, merges missing fields with defaults on write. `hydra state migrate` optional command (nice-to-have, not required).

## 11. Feature 7 — Benchmark Infrastructure

### 11.1 Location

**Separate repo:** `github.com/Zandereins/hydra-bench` (public, to be created as part of 2.0 work).

### 11.3 Scoring

**Two-stage:**

1. **Mechanical match:** Hydra finding is a candidate hit if `file` matches and `lines` overlap within ±5 of any expected finding.
2. **LLM-as-judge verification:** Candidate hits are routed to dual-judge (one Claude, one Codex). Both must independently agree the Hydra finding describes the same issue as the expected. Disagreement → manual review (queue written to `leaderboard-disputes.md`).

**Metrics per case:**
- **TP:** expected findings covered
- **FP:** Hydra findings with no expected match AND judged "noise" by dual-judge
- **FN:** expected findings not covered

**Aggregate metrics:**
- Precision = TP / (TP + FP)
- Recall = TP / (TP + FN)
- F1 = 2·P·R / (P+R)
- **Noise Ratio** = FP count / (KLOC of changed files)
- **Critical Recall** = Recall restricted to `cwe in {CWE Top 25} or severity == SERIOUS`

**FP Penalty:** Weighted F1 = (TP · 1.0 − FP_confident · 0.5) / (TP + FN). Prevents recall-gaming by carpet-bombing noise.

### 11.4 Reproducibility

Each case run **5 times** (different random seeds if supported, else just re-runs). Reports **median metric + variance (MAD)**. Cases with MAD > 15% of median → flagged "unstable", excluded from aggregate scores but still listed.

### 11.5 Latent-Bug Discovery Mode

Cases with `latent: true` in manifest:
- `diff_patch` is the code **before** the CVE-fix PR was merged
- `expected_findings` describes the bug that was later fixed
- Ground truth: Hydra should detect the bug despite being invoked on the "clean-looking" pre-fix code

**Intrinsic leakage protection:** cases are selected from PRs merged **after** the evaluated model's training cutoff.

### 11.7 Commands

```bash
hydra bench                       # run all cases, standard mode, default profile
hydra bench --case 001            # run single case
hydra bench --mode deep           # deep mode
hydra bench --profile budget      # all-sonnet
hydra bench --compare <run-id>    # compare current run vs. a stored past run
hydra bench --update              # pull latest bench repo
hydra bench --list                # show all cases
```

`--compare <run-id>`: each `hydra bench` invocation writes a run directory under `~/.cache/hydra/bench/runs/<timestamp>/` with aggregate metrics + per-case scores. `--compare <run-id>` runs the suite, then diffs metrics against the named past run. Useful for catching regressions between Hydra versions or after config changes. Run IDs are the timestamp directory names (auto-completed if shell supports it).

**Output:** `~/.cache/hydra/bench/runs/<timestamp>/report.md` with per-case scores, aggregate F1/Noise/Critical-Recall, and a leaderboard entry.

## 13. Testing Strategy

### 13.1 Unit (new)

- `test_grounding.py` — mechanical citation verification: file missing, range missing, token mismatch, token match, partial match threshold
- `test_skeptic.py` — skeptic prompt format, timeout handling, tool-call cap, PASS/FAIL/UNCLEAR routing
- `test_seed_report.py` — schema validation, tool output merging, graceful degradation when a tool missing
- `test_tensions.py` — correlation algorithm (position split, severity delta, cross-family split); tension record construction
- `test_model_resolver.py` — precedence chain, CLI flag parsing, config merge, chairman downgrade warning
- `test_echo.py` — 7-check protocol; no-PR-description edge case

### 13.2 Integration (new)

- `test_pipeline_standard.py` — end-to-end standard mode with mocked LLM responses; verify two-phase flow produces correct state.json v3
- `test_pipeline_deep.py` — same for deep mode
- `test_pipeline_degradation.py` — simulate tool missing (Semgrep absent), advisor timeout, skeptic timeout; verify graceful degrade

### 13.3 Bench-based (the real quality gate)

Hydra 2.0 must achieve **on the 10 starter bench cases** (standard mode, default profile):
- Recall ≥ 0.70 (catches ≥7 of 10 bugs in majority of runs)
- Critical Recall ≥ 0.80 (catches ≥4 of 5 SERIOUS/CWE-Top-25 bugs)
- Noise Ratio ≤ 3.0 FP per KLOC changed
- F1 ≥ 0.65
- **Regression guard:** no case's F1 may drop by more than 10% vs. a 1.x baseline

**1.x baseline capture procedure (must be first implementation task):**
Before any code change on this feature branch, check out `main@3506f93`, install that version as `hydra` in a scratch location, build the 10 bench cases (case definitions themselves are version-neutral), and run the suite 5× per case at temperature=0 with the default profile. Store aggregate + per-case metrics under `~/.cache/hydra/bench/baselines/1.x-2026-04-17/`. This directory is the immutable regression reference for the entire milestone.

Benchmarks are **the release gate**. No bench pass → no merge of 2.0.

### 13.4 Audit iterations (per Franz's ask)

The release process includes at least **3 full audit passes** before 2.0 merge:

1. **After Phase 2 implementation (mid-milestone):** multi-agent code review on the in-progress branch (Hydra on Hydra — dogfooding) + systematic-debugging skill if any bench regression
2. **After all features implemented (late-milestone):** full bench run × 5 + fresh-eyes audit by spawning 3 independent review subagents
3. **Pre-merge gate:** bench green across standard + deep modes; a final Hydra-deep-review on the PR diff; no SERIOUS findings unresolved

Each audit iterates with `superpowers:receiving-code-review` on feedback, not ignoring it.

## 14. Migration from Hydra 1.x

### 14.1 Breaking changes

- **Chain format in advisor JSON:** `{file_line, code_construct, assumption, failure_mode, impact}` → `{premise, execution_trace, conclusion}`. Advisor prompts rewritten. Old state.json v2 files read with backward-compat layer (fields migrated on first read).
- **Stranger advisor removed:** merged into Mies+. Any user-visible reference to Stranger (in 1.x reports) becomes "Mies+" in 2.0 reports. `--focus readability` still works; maps to Mies+ now.
- **`hydra tensions` / `hydra blind-spots` commands:** emit deprecation hints but still function in 2.0.

### 14.2 Non-breaking

- `hydra this`, `hydra this --deep`, `hydra this --focus X`: identical surface, new internals.
- Report format: adds `## Grounding Summary` and `## Tensions` sections but otherwise preserves existing structure.
- state.json v2 → v3: auto-migration on first read.

### 14.3 Opt-out of new behavior

For users wanting pure-1.x behavior (e.g., to compare output styles):
- `HYDRA_DISABLE_TWO_PHASE=1` — skip Phase 1, run flat fan-out as in 1.x
- `HYDRA_DISABLE_GROUNDING=1` — skip mechanical verification

These are **escape hatches, not features**. Documented in README troubleshooting section; removed in 3.0.

## 15. Success Criteria

Hydra 2.0 Core is released when **all** hold:

1. All 7 features implemented and unit-tested (§13.1, §13.2)
2. Bench metrics met on 10 starter cases (§13.3)
3. No SERIOUS findings from 3 audit iterations unresolved (§13.4)
4. README updated; `references/echo-protocol.md` and `references/skeptic-protocol.md` written
5. `hydra-bench` public repo created, populated with 10 cases, passing CI
6. Internal benchmark comparison vs 1.x shows: **no token regression**, F1 ≥ 1.x F1, Noise Ratio ≤ 1.x Noise Ratio
7. Manual end-to-end UAT by Franz: `hydra this` + `hydra this --deep` + `hydra this --tensions` + `hydra this --verify` + `hydra bench` all produce sensible output on a real PR/diff
8. PR merged to `main` with two Hydra self-reviews (one by 2.0-on-the-PR, one by 1.x baseline on a side-branch) showing 2.0 finds bugs 1.x misses without adding noise
