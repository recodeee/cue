# Profile Registry — DESIGN

> Status: **proposed** · Target: `cue` 0.10 · Author: ai-pair · Last updated: 2026-05-28
> Estimated effort: 1–2 days for the GitHub-as-registry MVP.

---

## Problem

Today, every profile a user builds is locked to their machine. `medusa-vite`, `designer-medusa-next`, custom shop variants — they live under `profiles/<name>/profile.yaml` on the author's disk and there's no way to share them. The two cohorts that suffer:

1. **Solo authors who use multiple machines.** Build a profile on laptop, want it on workstation. Today: manual copy.
2. **Team / community.** Build a profile that solves a stack-specific problem (Medusa-on-Vite-with-TanStack), want others to pick it up. Today: paste a YAML in a gist, hope the recipient knows where to put it.

Without sharing, every cue install is a fresh blank slate. Network-effects value is zero.

## Goals

- One-command publish: `cue share push <name>` uploads a profile.
- One-command install: `cue share install <user>/<name>` fetches and installs.
- Discovery: `cue share search <query>` lists matches with star/install counts.
- Updates: `cue share update [<name>]` pulls the latest version of installed shared profiles.
- Zero infra overhead. No registry server. No auth tokens beyond GitHub's.

## Non-goals (v1)

- Private profiles (everything public on day one — add private later via private repos).
- Skill / MCP sharing (profile references skills by ID — assume the recipient has them or `cue` can fetch them via the existing `cue discover install` path).
- Version pinning beyond git refs. No semver enforcement.
- Profile editing through a UI. Authors edit YAML locally and re-push.

## Architecture — GitHub as the registry

Use a single GitHub repo (`opencue/claude-code-skills-profiles`) as a flat-file registry. No backend, no database. Discovery via the GitHub Search API.

```
opencue/claude-code-skills-profiles/
├── README.md
├── index.json                  ← generated by CI on every merged PR
└── profiles/
    └── <user>/
        └── <name>/
            ├── profile.yaml    ← the user's profile, unmodified
            ├── README.md       ← optional; rendered on registry browse
            └── .meta.json      ← { author, stars, downloads, sha, updated_at }
```

**Why this works:**
- GitHub does the auth, storage, search, audit log, abuse handling, hosting.
- `index.json` is a single ~1MB file the CLI fetches once per session for offline-cap discovery.
- PR-gated submissions = lightweight human review before a profile lands.
- Star count on the directory = social signal.

## CLI surface

```
cue share push <name>                  # opens PR to add/update profiles/<user>/<name>/
cue share install <user>/<name>        # fetch + install to ~/.config/cue/shared/<user>/<name>/
cue share search <query>               # query index.json; rank by stars + recency
cue share list                         # show installed shared profiles + their commit sha
cue share update [<name>]              # bump installed shared profiles to latest sha
cue share remove <user>/<name>         # uninstall
```

**Flags:**
- `--private` (push) → adds the profile to a registry index for the user's own private repo instead of `opencue/claude-code-skills-profiles`. Deferred to v2.
- `--ref <sha|tag|branch>` (install) → pin to a specific git ref. Default: latest commit on `main`.
- `--dry-run` (push/install) → show what would happen, no side effects.

## Data flow

### Publish (`cue share push my-shop`)

```
   $HOME/.config/cue/profiles/my-shop/
                │
                ▼
   1. Validate locally (loadProfile + lint)
   2. Detect cue user identity (gh auth status → GitHub login)
   3. Clone or fast-forward $XDG_CACHE_HOME/cue/registry-fork/
   4. Copy profile.yaml + optional README.md → profiles/<gh-user>/my-shop/
   5. git commit + push to the user's fork (gh CLI handles)
   6. gh pr create against opencue/claude-code-skills-profiles
   7. Print PR URL
```

### Install (`cue share install jane/medusa-shop`)

```
   1. Fetch index.json (cached for 1h)
   2. Resolve {user}/{name} → { commit_sha, profile_yaml_url, deps }
   3. Download profile.yaml to ~/.config/cue/shared/jane/medusa-shop/
   4. Patch `name:` to "jane.medusa-shop" so it can't collide with built-ins
   5. Resolve required skills/MCPs:
       - Skills already on disk → no-op
       - Skills missing → reuse `cue discover install <skill>` flow
       - MCPs missing → print one-liner with the install command
   6. Print `cue use jane.medusa-shop` and a pin instruction
```

### Update (`cue share update`)

```
   For each profile in ~/.config/cue/shared/*/*/
     compare local .meta.json sha vs registry index.json sha
     if different → re-download + re-install
```

## File / directory layout

| Path | Role |
|---|---|
| `~/.config/cue/shared/<user>/<name>/profile.yaml` | The downloaded profile, namespaced |
| `~/.config/cue/shared/<user>/<name>/.meta.json` | `{ sha, installed_at, source_url }` |
| `~/.cache/cue/registry-index.json` | Cached `index.json`, refreshed hourly |
| `~/.cache/cue/registry-fork/` | User's fork checkout for push flow |
| `opencue/claude-code-skills-profiles/index.json` | `[{ author, name, description, stars, deps, sha, updated_at }]` |

## Profile namespacing

`user/name` becomes `user.name` when locally installed (e.g. `jane/medusa-shop` → `jane.medusa-shop`). This prevents:
- Shared profile colliding with a built-in profile of the same name.
- Two shared profiles from different authors with the same name colliding.
- Confusion about provenance — the `.` in the name is a visible "this is shared" cue.

`cue use jane.medusa-shop` works. Composites work: `cue use jane.medusa-shop+backend`.

## Dependencies (skills, MCPs)

A shared profile likely declares skills the recipient doesn't have. The install flow handles this transitively:

```yaml
# profiles/jane/medusa-shop/profile.yaml
name: jane.medusa-shop
description: Jane's opinionated Medusa shop stack
inherits: medusa-vite
skills:
  local:
    - design/awwwards-clone        # builtin — already on disk
    - jane/cart-drawer-pattern     # community skill — auto-fetch
mcps:
  - coolify                        # builtin
  - browserbase                    # community MCP — needs Smithery install
```

Install flow:
1. Resolve every skill/MCP id locally.
2. Missing builtins → user error ("update cue").
3. Missing community skills → `cue discover install jane/cart-drawer-pattern` (existing flow).
4. Missing MCPs → print one-liner: `cue marketplace install-mcp browserbase` (existing flow).

Never auto-install MCPs without user OK. Skills are lower-stakes; auto-fetch with `--yes` flag, prompt otherwise.

## Trust model

Shared profiles are **read-only** by default — they get materialized into `~/.config/cue/runtime/jane.medusa-shop/` like any other profile, but `cue use` won't run their hooks until the user confirms once per install:

```
$ cue use jane.medusa-shop
First use of shared profile jane.medusa-shop (sha: 7f3a9b1)
This profile declares 2 hooks: pre-commit, post-tool-use
Inspect: ~/.config/cue/shared/jane/medusa-shop/profile.yaml
[y] use once  [Y] use and trust forever  [n] cancel
```

Trust persists in `.meta.json`. Re-prompts on sha change.

## Open questions

- **GitHub-only or git-agnostic?** Starting GitHub-only because `gh` CLI handles auth uniformly. Git-agnostic is a v2.
- **What about people without `gh` installed?** Fall back to `curl` for read-only (install/search/update). Push requires `gh`.
- **Skill version pinning.** Today skills are referenced by id, no version. If `jane/cart-drawer-pattern` changes between her push and your install, you get her newest version. Long-term: pin skill commits in the profile's `.meta.json`.
- **License.** Default to MIT? Or require profile authors to declare? Probably MIT in v1, opt-out via `LICENSE` file in their profile dir.

## MVP scope (v1, what ships in 1–2 days)

✅ `cue share push` — opens a PR via `gh`. Manual review merges.
✅ `cue share install <user>/<name>` — fetches profile.yaml, namespaces, installs.
✅ `cue share search <q>` — substring match on cached index.
✅ `cue share update` — refresh installed profiles to latest sha.
✅ `cue share list` — show what's installed.
✅ Trust prompt on first `cue use` of a shared profile.

❌ Private registries (v2).
❌ Skill commit-pinning (v2).
❌ Discovery via stars/install-counts (data exists day one; UI sorts on it from the start).
❌ Web browsing UI (the dashboard — see `docs/design/dashboard.md`).

## Test plan

- Unit: index.json parsing + cache TTL.
- Integration (mocked `gh`): push flow writes correct files, opens PR.
- Integration (mocked HTTP): install flow downloads + installs + namespaces.
- Manual: end-to-end push → review → install on a second machine.

## Rollout

1. Create `opencue/claude-code-skills-profiles` repo with `index.json` empty array + README explaining contribution flow.
2. Build CLI commands behind a `--experimental` flag for the first week.
3. Author + push 5 starter profiles (medusa-shop-vite, designer-medusa-next, etc.) so the registry isn't empty.
4. Document on cue.so landing page.
5. Drop `--experimental` after the first 10 community submissions land cleanly.
